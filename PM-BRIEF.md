## What JoeQuest is

A recommendation engine that reads a coffee shop's Google reviews and tells you the ONE drink and ONE food item most worth ordering. Picks roll up into city-wide "best of" awards. Picks are generated by Claude (Opus 4.7) using JSON-schema-enforced structured output. As of now there's a working web MVP for Boca Raton, FL.

## Where the project is right now

We have a **live, deployed MVP**:
- **Public URL: https://joequest.onrender.com**
- **Source: https://github.com/Zshrem1974/joequest**

## Pathway we built (in ~/joequest/)

1. **test-offline.js** — Brain-only smoke test. Hardcoded reviews → per-café pick. Validated the prompt + JSON schema with no Google dependency.
2. **joequest-engine.js** — Single-café end-to-end. Google Places search → reviews → Claude pick (CLI).
3. **scan-list.js** — Multi-café batch scanner. Takes a curated list, runs each through the engine in parallel (concurrency 4), then a second curation pass picks the city-wide best drink + best food.
4. **recurate.js** — Cheap re-curation. Hardcodes already-collected per-café picks, runs only the final curation call. Lets us re-rank without re-paying for the scan.
5. **server.js** — Express MVP. `GET /api/cafes` lists the city's cafés, `GET /api/cafes/:id` returns AI picks on demand (lazily generated, cached 7 days). `GET /api/status` reports key configuration.
6. **public/index.html** — Vanilla-JS UI. Coffee-themed cards, modal detail view, skeleton loading state, confidence badges, reviewer quotes. Zero framework dependency.

## Architectural decisions worth knowing

- **Model:** claude-opus-4-7 with `output_config: { format: { type: "json_schema", schema: ... } }`. The API guarantees valid JSON matching the schema — no defensive parsing needed.
- **The prompt's anti-hallucination guardrails are load-bearing.** It forces: (a) only recommend items literally named in reviews, (b) supporting quote must be verbatim, (c) return `null` + confidence "none" when data is too thin. Without these, the model invents menu items.
- **Two-stage curation:** Per-café picks → city-wide curator. Curator weighs confidence > specificity > mention count > distinctiveness.
- **Ranking signal:** Cafés are sorted by `rating² × log(reviews)` — quality squared, weighted by review volume. Keeps 4.9-star ghost spots from outranking 4.5-star institutions.
- **Keys server-side only.** The browser never sees `GOOGLE_PLACES_API_KEY` or `ANTHROPIC_API_KEY`. This is non-negotiable for any path forward.
- **Caching:** In-memory Map with TTL. Fine for one-instance demo. Anything multi-instance needs Redis or a DB.

## Known limitations / known unknowns

- **Chain filter is naive.** Substring match only — Pura Vida Miami slipped through the live MVP because it's not on the blocklist. We need either a richer blocklist, a category filter (Places returns `types[]`), or a small LLM "is this a real specialty café?" gate.
- **Google Places returns max ~5 reviews per place.** That's the input ceiling. The model is working off a thin sample, which is why mention counts cap low.
- **Some text queries resolve to nearby-city locations.** First scan pulled Coconut Creek, Delray, and Deerfield places. The server uses `searchText` against a city string but doesn't enforce city boundaries — need a lat/lng bounding box or a post-filter on `formattedAddress`.
- **In-memory cache resets on server restart.** Need Postgres or Redis for durability.
- **"Confidence: high" with only 2 mentions still happens.** The rubric weights specificity heavily, which can over-rank a single emphatic review.
- **No reviewer attribution.** We quote reviewers but don't link back to the source. Trust signal is weaker than it could be.
- **No multi-city.** Hardcoded to Boca. Generalizing means a city picker + region-specific curation lists.

## Questions for the PM to decide

Now that we have a live demo, the open questions sharpen:

1. **Surface & target user.** Web app for tourists/locals? Native mobile? Instagram-style city guide? Embedded widget for hotel concierge sites? API for partners (Yelp, Resy, DoorDash)? This decision drives everything downstream — design, distribution, monetization.
2. **City expansion model.** Hand-curated rollout (one city/week) vs. self-serve discovery (user types any city, we run a fresh scan). The first is defensible; the second is scalable. Maybe both — paid "official" cities + on-demand exploration.
3. **Data freshness.** Re-scan nightly? Weekly? On-demand via "refresh" button? Each strategy has a cost profile.
4. **The moat.** Anyone could replicate the current engine in a weekend. The defensible piece is probably one of: (a) curation taste (we filter & re-rank in ways Google won't), (b) review aggregation beyond Google's 5-review cap (TripAdvisor, Yelp, Reddit, IG comments), (c) submitted/community reviews layer, (d) deep integrations with cafés themselves (verified menus, "order this" buttons).
5. **Monetization.** Affiliate links to order (DoorDash, Toast, Square)? Paid placement risks killing the trust signal. Subscription for premium cities? Sponsored "discover" features? Café-paid verified listing tier?
6. **Trust signals.** Today: a reviewer quote + confidence badge. Considered: linking to source reviews, showing reviewer count threshold, "last updated" timestamp, "verified by [N] users" social proof, photos of the recommended item.
7. **Onboarding hook.** What's the 5-second pitch on first visit? "Tell us a city, we'll tell you what to order" feels stronger than "browse cafés." Consider a /quest landing route that asks "where are you right now?" and routes to the best single café + its pick.
8. **Beyond drink + food.** Should we also pick best time to visit? Best seat? Best for WFH vs. date vs. solo morning? Each axis is another Claude call but also another reason to come back.

## How to run it

**Production (already deployed):** https://joequest.onrender.com

**Locally:**
```bash
cd ~/joequest
GOOGLE_PLACES_API_KEY=xxx ANTHROPIC_API_KEY=yyy node server.js
# → open http://localhost:3000
```

**Deploy a new instance:** Push to a GitHub repo, connect it as a Render Blueprint
(reads `render.yaml`), add the two API keys as env vars in Render's dashboard. Full
guide in `DEPLOY.md`. Fly.io alternative also configured in `fly.toml`.

Code is plain ES modules + Express, no framework, no build step.
